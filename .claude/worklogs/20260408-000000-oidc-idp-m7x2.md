## 1. research

**Goal:** Design and plan the migration of homelab-auth-service from a proprietary JWT login API to a fully spec-compliant OIDC Identity Provider using Spring Authorization Server, enabling SSO for all furchert.ch homelab services (Grafana, Home Assistant, etc.).

**Current state:**
- Spring Boot 4.0.5 / Spring Security 7.0 / Java 25
- Custom JWT auth via jjwt 0.12.6: `POST /api/v1/auth/login`, `/refresh`, `/logout`, `GET /api/v1/auth/jwks`
- RSA key pair at `src/main/resources/keys/private.pem` / `public.pem`, key ID `auth-service-v1`
- `User` entity with `username`, `email`, `passwordHash`, `role`, `status`
- `CustomUserDetailsService`, `RsaKeyProvider`, `RsaKeyProperties` all reusable
- No existing consumers of the auth API
- 40 unit tests green (0 integration tests currently passing — Docker socket issue locally)

**Repo areas inspected:**
- `config/SecurityConfig.java` — stateless, custom JWT filter, single chain
- `security/JwtService.java` — jjwt token generation, `KEY_ID = "auth-service-v1"`
- `service/AuthService.java` — login/refresh/logout/JWKS logic
- `service/TokenCleanupScheduler.java` — hourly purge of `refresh_tokens`
- `entity/RefreshToken.java` — SHA-256 hashed tokens in DB
- `controller/UserController.java` — uses `@AuthenticationPrincipal String username`
- `test/controller/UserControllerTest.java` — uses `@MockitoBean JwtService`, `.with(user(...))`
- `pom.xml` — no OAuth2/OIDC dep, no Thymeleaf
- `src/main/resources/db/migration/V1, V2` — existing Flyway migrations

**Assumptions:**
- A1: No services currently consume `/api/v1/auth/**` — confirmed by user [high]
- A2: Single-pod K3s deployment — sessions can stay in-memory [high]
- A3: Client secrets will be pre-hashed BCrypt in env vars [high]
- A4: `spring-boot-starter-oauth2-authorization-server` is compatible with SB 4.0.5 [high — BOM-managed]

**Open questions:**
- Q1: Spring AS version compatibility with SB 4.0.5 — resolved: use `spring-boot-starter-oauth2-authorization-server` (BOM-managed) [non-blocker]

**Research summary:**
- Spring Authorization Server can reuse existing `RsaKeyProvider` for `JWKSource` — same RSA keys, same `kid`
- `JdbcOAuth2AuthorizationService` does NOT auto-purge expired rows — `TokenCleanupScheduler` must be rewritten, not deleted
- `UserController.@AuthenticationPrincipal String username` breaks with JWT resource server — must change to `@AuthenticationPrincipal(expression = "subject")`

## 2. plan

**Goal:** homelab-auth-service exposes a fully spec-compliant OIDC IdP such that any service supporting OAuth2/OIDC Authorization Code Flow can use it for SSO without custom code.

**Design spec:** `docs/superpowers/specs/2026-04-07-oidc-design.md`
**Implementation plan:** `docs/superpowers/plans/2026-04-08-oidc-idp.md`

**Options considered:**

| Option | Decision |
|---|---|
| Spring Authorization Server (BOM-managed starter) | **Chosen** — spec-compliant, battle-tested, reuses existing infra |
| Custom OIDC endpoints on top of jjwt | Rejected — reimplements Spring AS with higher risk |
| Minimal OIDC subset for Grafana only | Rejected — breaks for future services, not future-proof |

**Key decisions locked in:**
- Two filter chains: AS (order 1) + resource server (order 2)
- `spring-boot-starter-oauth2-authorization-server` — BOM-managed, version resolved by SB 4.0.5
- `spring-boot-starter-thymeleaf` — for login form
- Static client registration via `app.oidc.clients[]` in `application.yaml`
- Client secrets: pre-hashed `{bcrypt}$2a$10$...` in env vars — no runtime hashing
- RP-Initiated Logout via `logoutEndpoint(Customizer.withDefaults())`
- `requireAuthorizationConsent(false)` on all clients — first-party apps
- `requireProofKey(true)` on all clients — PKCE mandatory
- `JdbcOAuth2AuthorizationService` + Flyway V3 for token persistence
- `TokenCleanupScheduler` rewritten to purge `oauth2_authorization` table
- `@AuthenticationPrincipal(expression = "subject")` fix in `UserController`
- `server.forward-headers-strategy=native` for ingress support

**Tasks:** 12 tasks in `docs/superpowers/plans/2026-04-08-oidc-idp.md`

**Code review findings incorporated:**
1. RP-Initiated Logout (`/connect/logout`) added
2. Consent screen bypass (`requireAuthorizationConsent(false)`)
3. BCrypt pre-hashing (env vars contain hashed value, no runtime cost)
4. Single-pod session limitation documented as risk in DEPLOYMENT.md
5. CSRF chain separation: resource server chain scoped to `/api/v1/**` only

**Self-review findings fixed:**
- `TokenCleanupScheduler` rewritten (not deleted) — JDBC store has no auto-purge
- `RsaKeyProperties` reused for token lifetime config in Spring AS `TokenSettings`
- `@AuthenticationPrincipal String username` → `@AuthenticationPrincipal(expression = "subject")` in `UserController`
