## 1. research

**Goal:** Understand why `GET /oauth2/authorize?response_type=code` returns 400 `[invalid_request] OAuth 2.0 Parameter: response_type` instead of 302 redirect to `/login`.

**Current state:**
- 2 of 48 tests fail in CI: `authorizeEndpointRedirectsToLoginWhenUnauthenticated` and `fullAuthCodeFlowIssuesTokens` in `OidcFlowIntegrationTest`
- All other 46 tests pass, including OIDC discovery, JWKS, token rejection, admin auth, login page, and logout
- The authorization server infrastructure is working (discovery returns correct endpoints)
- These tests were never run in CI before (required Docker for Testcontainers)

**Repo areas inspected:**
- `src/test/java/.../integration/OidcFlowIntegrationTest.java` — tests use `mockMvc.perform(get("/oauth2/authorize").param("response_type", "code")...)`
- `src/main/java/.../config/AuthorizationServerConfig.java` — AS chain with manual `securityMatcher("/oauth2/**", ...)`
- `src/main/java/.../config/SecurityConfig.java` — login chain, API chain, catch-all chain
- `src/test/resources/application.yaml` — test config with JWT keys and Flyway settings
- `src/test/java/.../AbstractIntegrationTest.java` — `@DynamicPropertySource` providing OIDC client config

**Git history notes:**
- Commit `f62f67c` — most recent on `oidc-switch` branch, split security chains
- Commit `84a7fb4` — replaced JWT auth with OIDC Identity Provider

**Assumptions:**
- A1: Spring AS reads `response_type` from `request.getQueryString()`, not `request.getParameterMap()` [high — confirmed by Spring AS own tests using `updateQueryString()` helper]
- A2: MockMvc's `.param()` adds to parameter map but does NOT set query string for GET requests [high — confirmed by CI logs showing params in map but AS rejecting them]

**Open questions:**
- Q1: Will putting params in the URL query string fix both tests? [non-blocker — highly likely based on root cause]

**Research summary:**
- Spring AS's `OAuth2AuthorizationCodeRequestAuthenticationConverter` reads authorization request params from `request.getQueryString()` (via `OAuth2EndpointUtils.getQueryParameters()`), not from `request.getParameterMap()`
- MockMvc's `.param()` method adds to the request's parameter map but does not populate the query string for GET requests in Spring Framework 7.0
- Spring AS's own unit tests explicitly call `updateQueryString(request)` after `addParameter()` to ensure the query string matches the parameter map (confirmed via GitHub source inspection)

## 2. plan

**Goal:** Both `OidcFlowIntegrationTest` methods pass in CI by constructing authorize requests with parameters in the URL query string.

### Context

**Summary:** Spring Authorization Server's authorization endpoint reads OAuth2 parameters from the HTTP query string, not from the servlet parameter map. MockMvc's `.param()` for GET requests populates only the parameter map. The fix is to construct the authorize URL with query parameters using `UriComponentsBuilder`, ensuring Spring AS can read them from the query string.

**Assumptions:**

| ID | Assumption | Confidence | Implication if wrong |
|----|------------|------------|----------------------|
| A1 | Spring AS reads from query string via `getQueryParameters()` | high | Would need to debug AS filter chain further |
| A2 | `UriComponentsBuilder` query params appear in MockMvc's query string | high | Would fall back to raw URL string concatenation |

**Open questions:**

| ID | Question | Severity |
|----|----------|----------|
| Q1 | Does the full auth code flow (test 2) need additional MockMvc adjustments beyond step 1? | non-blocker — step 3 re-GETs the saved authorize URL which should already have query params |

### Options considered

**Option 1 - Use `UriComponentsBuilder` to build authorize URL (chosen)**
- What: Replace `.param()` calls with a `UriComponentsBuilder` that produces a URL with query string, then pass that URL to `mockMvc.perform(get(url))`
- Pros: Clean, handles URL encoding automatically, idiomatic Spring approach
- Cons: Slightly more verbose than `.param()`

**Option 2 - Manually concatenate query string (rejected)**
- What: Build the URL string manually with `+` operator and `URLEncoder.encode()`
- Pros: Simple, no extra imports
- Rejected because: Error-prone for URL encoding, harder to read, `UriComponentsBuilder` is more maintainable

**Option 3 - Use `MockMvcRequestBuilders.get(URI)` with `URI.create()` (rejected)**
- What: Build a `java.net.URI` with query parameters
- Pros: Standard Java API
- Rejected because: `UriComponentsBuilder` is the Spring-standard way and integrates better with MockMvc

### Changes

**C1 - Fix authorize request construction in both failing tests**
Files: `src/test/java/ch/furchert/homelab/auth/integration/OidcFlowIntegrationTest.java`
